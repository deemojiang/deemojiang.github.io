## 1. 文档目的与范围

### 1.1 目的

明确「企航联盟平台」的产品边界、用户角色、核心功能与数据口径，作为产品设计、研发联调、监管汇报与迭代规划的依据。

### 1.2 范围

- **本期范围**：平台由**两大功能块**构成——**大屏数据展示**、**管理后台**；二者共用同一套组织与权限主数据，大屏按角色展示全县或本企业范围的数据。
- **大屏**：首页三大入口（舆情导控、侵权线索、网络安全），支持**二级穿透**（模块专题页）、**三级穿透**（单条数据详情）。
- **管理后台**：以**权限与组织管理**为核心，含**平台管理员**、**企业账号**两类主体能力，以及**企业入驻**流程；可扩展数据源配置、审计查询等运维能力。
- **不在本期默认范围**（除非单独立项）：非联盟企业的数据接入、跨省数据共享、具体硬件型号采购与施工。

## 2. 背景与目标

### 2.1 背景

县域内多家企业加入「企航联盟」，监管部门（网信办、公安网警、市场监督管理局、数据局等）需要对联盟内企业的**舆情态势**、**涉企侵权线索**与**网络安全态势**进行统一感知与协同处置；龙头企业（如天能、超威等）需要**仅查看与本企业相关**的数据，部分数据由监管部门提供，部分由企业自有系统对接。

### 2.2 产品目标

1. **监管侧**：一屏掌握全县（联盟内）企业在舆情、侵权、安全三方面关键指标与告警线索，支撑值班与指挥场景。
2. **企业侧**：授权企业用户登录后，数据严格隔离，仅展示本企业授权范围内的指标与事件。
3. **数据可信**：大屏与业务页面需能区分或标注**数据来源**（网信办、阿里侵权线索平台、第三方安服、**网络空间测绘类平台（如 FOFA）**、**威胁情报类能力（如微步在线 API/情报服务）**、企业出口设备、企业自有系统等），避免口径混淆。

### 2.3 成功度量（建议，待业务确认）

| 指标 | 说明 |
|------|------|
| 数据覆盖 | 联盟内企业接入比例、核心数据源对接完成数 |
| 时效 | 各主题数据从产生到可展示的 SLA（如 T+0 / 小时级） |
| 可用性 | 大屏关键页面可用率、重大活动保障预案 |
| 安全合规 | 权限误开放事件为 0；操作与导出留痕完整 |

## 3. 术语与缩写

| 术语 | 说明 |
|------|------|
| 联盟 / 入盟企业 | 已签约并接入企航联盟平台的企业 |
| 大屏展示端 | 面向指挥室、值班室的大屏可视化终端；支持一级首页、二级专题、三级详情穿透 |
| 一级首页 | 大屏入口页，并列展示舆情导控、侵权线索、网络安全三大板块（可含全县或本企业总览 KPI） |
| 二级穿透 | 点击某一板块后进入该板块的专题数据页（列表、图表、分区看板） |
| 三级穿透 | 在二级页点击单条记录进入详情抽屉/全屏详情页 |
| 平台管理员 | 管理后台最高业务权限账号，可查看并管理全县联盟在平台上的全部数据与配置（可按实际拆分为多部门子管理员，见 §4.3） |
| 企业账号 | 绑定某一入盟企业的登录账号，大屏与管理端仅可见该企业在舆情、侵权、安全三块的全部接入数据 |
| 舆情导控 | 与品牌、舆论、敏感信息相关的监测与引导相关能力（具体以网信办业务定义为准） |
| 涉企侵权线索 | 侵害企业合法权益的线索，本期强调市监局侧与阿里涉企侵权线索平台对接 |
| 出口侧设备 | 部署在企业网络出口侧的安全或探针类设备，用于流量/威胁可见性（具体型号与合规要求单独立项） |
| FOFA（网络空间测绘） | 面向全球互联网可达资产的搜索引擎类能力，用于**暴露面盘点**与资产指纹识别（运营方：华顺信安等，以采购合同为准） |
| 微步在线（威胁情报） | 以 IOC/信誉/关联分析为主的**威胁情报**能力，用于**恶意性研判**与事件溯源 enrich（产品含情报 API、TIP/TDP 等，以实际采购为准） |

## 4. 用户角色与权限模型

### 4.1 核心角色（与管理后台一致）

| 角色 | 数据可见范围（大屏 + 后台业务视图） | 管理后台能力 |
|------|--------------------------------------|--------------|
| **平台管理员** | **全县入盟企业**在舆情、侵权、安全三块下的全部接入数据；可按全县聚合与按企业下钻 | 权限与组织管理、企业入驻审核、账号生命周期、系统级配置（可扩展数据源开关等）；**可查看平台侧全部信息** |
| **企业账号** | **仅本企业**在舆情导控、侵权线索、网络安全三块下的全部接入数据（指标与字段同全县版，范围裁剪为本企业） | 本企业资料维护、**子账号管理（可选）**、入驻材料补充；更多配置项由管理员策略决定是否开放 |

### 4.2 权限原则

1. **组织隔离**：企业账号在任何终端仅能访问绑定组织的数据；平台管理员不受单企业限制。
2. **三块对齐**：同一角色在三个业务板块沿用同一套数据范围规则（全县 vs 本企业），避免跨板块口径不一致。
3. **最小必要与脱敏**：三级详情中的敏感字段按角色与法务要求脱敏；操作留痕（详见第 8 节）。

### 4.3 与监管多部门的关系（可选扩展）

网信办、网警、市监局、数据局等可对应**一个或多个平台管理员账号**，或采用**管理员组 + 数据/菜单策略**。业务上仍满足「监管侧看全县、企业看本家」；本文以 **平台管理员 / 企业账号** 为权限建模主线条，便于研发实现。

## 5. 产品总体架构

### 5.1 两大子系统

| 子系统 | 定位 | 主要用户 |
|--------|------|----------|
| **大屏数据展示** | 值班、指挥、汇报场景；**一级首页 → 二级专题 → 三级详情** | 平台管理员（全县视角）、企业账号（本企业视角） |
| **管理后台** | 权限与组织、企业入驻、账号与审计 | 以平台管理员为主；企业账号能力见 §6.2 |

### 5.2 业务板块（三大块）

舆情导控、侵权线索、网络安全在大屏首页并列呈现；管理后台按组织查看业务数据时，亦按此三块组织菜单或 Tab。

## 6. 功能需求详述

### 6.1 大屏数据展示

#### 6.1.1 穿透层级（统一交互）

| 层级 | 名称 | 行为 | 通用展示要求 |
|------|------|------|----------------|
| **一级** | **首页** | 三大板块入口：**舆情导控**、**侵权线索**、**网络安全**；每块展示**少量 KPI 摘要**（如近 24h/7 日关键数字、状态灯） | 布局对称；可配置「默认高亮板块」与**首页轮播**；显式展示**数据截止时间**；**平台管理员**见全县汇总，**企业账号**见本企业汇总 |
| **二级** | **板块专题页** | 点击某一板块进入；展示该板块**全部可视化数据**（多区块组合），即「二级穿透」 | 顶部：**筛选**（时间、企业——管理员可用、来源类型等）+ 面包屑返回首页；中部：**图表区 + 列表区**（及地图/关系图，数据具备时）；列表或图中每一项可进入三级 |
| **三级** | **单条详情** | 点击二级页中的**单条**事件/线索/告警/资产/IOC 等，即「三级穿透」 | **全屏页或右侧抽屉**：完整字段、**来源标签**、时间线/处置节点（如有）；**返回二级**且尽量保留二级筛选状态 |

**导航**：非首页需有「返回上一级」「回到首页」。

---

#### 6.1.2 舆情导控 — 可展示数据与展示方式

**数据来源**：县网信办（主）、入盟企业补充（次）。

| 穿透 | 展示内容 | 建议展示形式 | 数据范围 |
|------|----------|--------------|----------|
| **一级** | 舆情**事件总数**（周期可配）、**待处置数**、**敏感级别分布**、**近 7 日趋势** | 三卡片 KPI + 迷你趋势线；可选「重点企业」小卡片（**后台可配置**，禁止写死企业名） | 管理员：全县；企业：本企业 |
| **二级** | **事件列表**：时间、关联企业（管理员列表含企业列）、标题/摘要、敏感级别、来源（网信办/企业）、状态 | 左/上：**趋势或级别堆叠图**；右/下：**可排序、可分页表格**；支持按来源、级别、时间筛选 | 企业账号列表仅本企业 |
| **二级** | **传播或声量概览**（有渠道数据时）：渠道占比、热词 | 环形图、词云 | 同上 |
| **三级** | **单条舆情详情**：正文或摘要全文、首发时间、传播链路摘要、关联链接、标签、**处置记录**、数据来源 | 纵向时间线 + 文本区；外链受控打开 | 详情敏感信息脱敏 |

---

#### 6.1.3 侵权线索 — 可展示数据与展示方式

**数据来源**：阿里涉企侵权线索平台（市监局侧）、企业自有线索。

| 穿透 | 展示内容 | 建议展示形式 | 数据范围 |
|------|----------|--------------|----------|
| **一级** | **线索总量**、**在办/已办结**、**近 30 日新增**、**来源占比**（平台/企业） | KPI 卡 + 来源占比条形图 | 企业：仅本企业为权利人或被侵权主体的线索 |
| **二级** | **线索列表**：编号、类型（商标/专利/假冒伪劣等）、权利企业、摘要、来源、登记时间、状态、经办方（脱敏） | 表格 + **阶段漏斗或状态条形图**；按类型、状态、时间筛选 | 管理员可按企业筛选 |
| **二级** | **地域或行业分布**（字段具备时） | 地图或柱状图 | 可选区块 |
| **三级** | **单条线索详情**：事实描述、证据材料索引、法律依据摘要、**处置节点**（受理、调查、反馈）、关联主体 | 时间线 + 结构化字段；证据以「已归档/受控下载」等形式按权限开放 | 企业仅看与己相关线索 |

---

#### 6.1.4 网络安全 — 可展示数据与展示方式

**叙事**：暴露面可见 → 威胁可判 → 处置可跟。

**能力分层与数据来源（摘要）**：

| 分层 | 数据内容 | 典型来源 |
|------|----------|----------|
| 网络基础设施安全 | 态势评分、告警汇总、关键设施状态 | 监管/数据局口径 + 企业上报 |
| 第三方安全服务 | 扫描任务、高危漏洞数、渗透/演练结论摘要 | 第三方服务商 API/报告 |
| 企业出口侧设备 | 设备在线率、策略命中、流量告警摘要 | 企业设备回传 |
| 暴露面测绘 | 公网 IP/端口、服务指纹、证书、站点 title 等 | **FOFA（或同类）API** |
| 威胁情报 | IOC 信誉、威胁类型、PDNS/关联样本摘要等 | **微步情报 API（或 TIP）** |

| 穿透 | 展示内容 | 建议展示形式 | 数据范围 |
|------|----------|--------------|----------|
| **一级** | **综合风险或高危告警数**、**未修复漏洞数**、**公网暴露资产数**、**情报标记高危数**、**出口设备在线率**（指标可配置） | 多 KPI +风险灯；指标旁 **数据来源图标** | 管理员：全县；企业：本企业 |
| **二级** | **分区**：基础设施/告警、第三方扫描、出口设备、**测绘资产**、**情报 enrich** | 每区 **小 KPI + 表格**；告警与资产表可联动「仅高危」；测绘行：IP、端口、指纹摘要、测绘时间；情报行：IOC、判定、等级、更新时间 | 企业无其他企业明细 |
| **二级** | **全县企业安全排行**（管理员） | 横向条形图 | 企业账号可改为「本企业历史对比」 |
| **三级** | **单条告警/漏洞/资产/IOC 详情**：原始字段、来源、建议处置、关联跳转（如资产→情报） | 可选 **联合时间线**（测绘特征 + 情报标签 + 内部告警） | 测绘结果文案注明「公网探测事实≠已入侵」 |

---

#### 6.1.5 网络安全外部源：FOFA 与微步（能力边界）

| 外部源 | 回答的问题 | 对接与展示要点 |
|--------|------------|----------------|
| **FOFA** | 公网侧**暴露了哪些资产、服务指纹是什么** | 仅查询**已入驻企业申报**的域名/IP 段；API 定时拉取、平台侧聚合；三级详情展示 API 返回字段（IP、端口、title、证书、指纹等） |
| **微步在线** | 某 IP/域名/IOC **是否涉恶意、标签与关联证据** | 告警与测绘结果提交情报 API enrich；三级详情展示威胁类型、信誉、关联摘要等（以开通接口为准）；可与 FOFA 高危资产联动 enrich |

**合规**：测绘与情报查询全量留痕；个人隐私类 Whois 等按法务脱敏；遵守采购协议与数据出境要求。

---

#### 6.1.6 大屏通用补充

- **登录态**：大屏 URL 绑定账号或使用统一登录；未登录场景是否允许「脱敏公众版」由实施决定，默认需登录。
- **轮播**：首页可自动轮询三大板块 KPI（可关闭）。
- **刷新**：各板块显示**最近同步时间**；后端定时任务 + 前端轮询间隔可配置。
- **布局**：适配常见大屏分辨率；亮色/暗色主题（UI 设计阶段定稿）。

### 6.2 管理后台

#### 6.2.1 定位

管理后台**以权限与组织管理为核心**，保证「**平台管理员掌握全县**、**企业账号只看本企业**」在全站一致落地，并支撑**企业入驻**闭环。业务数据展示与大屏**同源 API**（见 §7）。

#### 6.2.2 平台管理员功能

| 功能模块 | 说明 |
|----------|------|
| **管理员账号** | 创建/停用/重置密码；可预留「超级管理员」与细分操作权限 |
| **企业入驻** | 申请列表、**审核通过/驳回**与原因；通过后生成**企业组织**与默认**企业账号**（或邀请企业首账号） |
| **企业信息** | 维护企业名称、统一社会信用代码、对接人、**授权资产范围**（域名/IP 段，供安全测绘）等 |
| **企业账号管理** | 在企业下创建/停用子账号；角色：企业主账号 / 普通成员（可选） |
| **权限策略** | 维护角色与数据范围（平台管理员 vs 企业账号）；后续可扩展菜单级权限 |
| **业务数据浏览（可选）** | 以列表查看三块业务数据（同大屏二级/三级字段），便于核验；**导出需审计** |
| **审计与日志** | 登录日志、关键操作（审核、改权限、导出）留痕 |

#### 6.2.3 企业账号功能（管理后台）

| 功能模块 | 说明 |
|----------|------|
| **本企业信息** | 查看/补充入驻资料、对接人、授权资产声明等 |
| **子账号（可选）** | 邀请成员、重置密码 |
| **业务数据** | Web 列表查看本企业三块数据（等价大屏二、三级，便于办公电脑使用） |

#### 6.2.4 企业入驻流程

1. 企业提交入驻申请（主体信息、联系人、意向板块等）。
2. 平台管理员审核；可要求补材料。
3. 审核通过：写入组织主数据、开通企业账号、触发外部数据源侧企业标识同步（如有）。
4. 驳回：企业可修改后再次提交。

建议状态：**待审 / 补正中 / 已通过 / 已驳回**（实施时可微调命名）。

#### 6.2.5 后续可扩展（非本期必做）

数据源连接配置、大屏轮播策略、告警阈值、与监管侧单点登录等。

## 7. 大屏与管理端一致性

- **数据同源**：大屏与管理端业务列表共用 API，避免两套统计口径。
- **权限同源**：请求统一携带组织上下文；企业账号 `org_id` 固定，平台管理员具备全县查询能力。
- **详情对齐**：三级详情与后台详情页字段一致（后台可增加「内部备注」类字段，依赖额外权限位）。

## 8. 数据与非功能需求

### 8.1 数据质量与口径

- 统一**主数据**：企业标识（统一社会信用代码或平台企业 ID）、组织树、角色绑定。
- 对外展示需标注**数据来源**与**统计截止时间**。

### 8.2 安全与合规

- 传输与存储加密、访问控制、**操作审计**（谁在看、是否导出）。
- 个人信息与案件细节：**脱敏规则**由法务与业务部门确认。
- 日志留存周期符合当地监管要求。

### 8.3 性能

- 大屏首屏加载时间目标（建议：首屏关键内容 ≤ N 秒，N 由项目约定）。
- 高并发只读场景下的缓存与降级策略（数据源不可用时的占位与提示）。

### 8.4 可靠性

- 关键数据源异常时的告警（运维可见）。
- 重大活动保障：预案与值班接口人（流程层面）。
