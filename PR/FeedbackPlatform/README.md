## 2. 项目概述

### 2.1 背景与目标

建设一套面向居民的物业问题上报与政务协同办理平台：居民通过手机扫码即可提交问题；由住建局统一分流、分派或自办；乡镇/街道管理员按权限处理问题并闭环；管理侧支持移动端与电脑端协同，电脑端提供数据统计与监管视图。

### 2.2 产品形态

平台由 **三端** 组成：

| 端 | 设备形态 | 核心用户 | 核心目标 |
|----|----------|----------|----------|
| 用户端 | 手机（浏览器/H5/小程序等，实现方式待定） | 居民/报修人 | 扫码进入、填写并提交问题反馈 |
| 管理员手机端 | 手机 | 住建局管理员、乡镇/街道管理员 | 登录后分派或办理问题（与电脑端功能对齐，交互适配小屏） |
| 管理员电脑端 | 桌面浏览器 | 住建局管理员、乡镇/街道管理员 | 与手机端管理功能一致，并补充 **数据统计与报表** |

## 3. 角色与权限

### 3.1 角色定义

| 角色 | 说明 | 主要能力 |
|------|------|----------|
| **普通用户（居民）** | 无需登录 | 扫码进入反馈页、提交问题（含图片等） |
| **住建局管理员** | 需登录 | **分流**：可将工单**分派至街道**（选项为 **龙山街道、雉城街道、画溪街道、太湖街道** 四者全称，下拉单选），或**自行办理**；查看所辖范围数据与统计 |
| **乡镇/街道管理员** | 需登录 | **办理**指派到本街道的问题：填写办理意见、选传图片，办结（可见范围与账号「管辖街道」配置一致，见 **3.4**） |

### 3.2 账号规模（一期）

- 管理端账号共 **5 个**（可配置：如 1 个住建局账号 + 4 个乡镇/街道账号，具体归属以业务确认表为准）。
- 登录凭证形式：**账号 + 密码**；须支持通过 **短信验证码** 的 **忘记密码/重设密码** 流程，见 **5.7**。

### 3.3 权限矩阵（摘要）

| 功能 | 普通用户 | 住建局管理员 | 乡镇/街道管理员 |
|------|:--------:|:------------:|:----------------:|
| 提交问题反馈 | ✅ | — | — |
| 查看全部工单 | — | ✅（按产品规则限定范围） | 仅本乡镇相关工单 |
| 分流/分派至街道 | — | ✅ | — |
| 自行办理（住建局） | — | ✅ | — |
| 办理/办结（乡镇） | — | — | ✅ |
| 管理端数据统计（电脑端） | — | ✅（建议） | 按权限可见本乡镇或全局，**以实现时配置为准** |
| 账号管理（增删改管理员、重置密码等，电脑端） | — | ✅（系统管理员/授权角色，以权限为准） | — |

> **说明**：若需「街道管理员仅看本街道」与「住建局看全市」的统计口径差异，在实现阶段以角色配置为准；本文档按「分流目标为四街道 + 住建局自办」描述业务。

### 3.4 账号管理（管理员 · 建议仅电脑端）

- **入口**：管理端 **电脑** 提供 **「账号管理」** 模块（与首页、工单列表等并列；手机端以「仅查看或跳转 Web」处理，不强制在手机上完成维护）。
- **能力**：
  - **新增 / 编辑** 管理端账号；填写或维护 **登录账号、姓名、手机号**、**角色**（如 **系统管理员**、**乡镇/街道管理员** 等，以实现为准）。
  - **乡镇/街道管理员** 须配置 **管辖街道**（**单选**，仅能从 **龙山街道、雉城街道、画溪街道、太湖街道** 四者中选其一，**不含「其他」**）。
  - **重置密码**：触发后向该账号 **绑定手机** 发送 **短信**（含临时密码或重设链接，**技术实现与安全策略** 在实现阶段定稿）。
- **与业务数据关系**：居民在反馈表单中「所在街道」可选 **其他**；**分派/转交、账号管辖街道** 等管理侧目标 **仅上述四街道**，**不包含「其他」**，与原型一致。

### 3.5 分派与处置通知

- 住建局对工单完成 **分派至某街道** 后，系统应向 **该街道对应管理员在平台登记的手机号** 发送 **短信通知**，提醒其 **登录处置**（是否同步站内消息、微信等由二期规划；短信文案需合规）。

## 4. 核心业务流程

### 4.1 居民上报流程

1. 用户扫描入口二维码，进入 **问题反馈界面**（用户端）。
2. 填写表单并提交。
3. 系统生成一条 **问题反馈工单**（或案件），节点进入 **待分派**（见第 6 节）。
4. 用户在提交成功页可仅做 **成功提示**；本期不要求用户端查询历史工单（若需后续迭代可补充）。

### 4.2 住建局分流流程

1. 住建局管理员登录管理端（手机或电脑）。
2. 查看待分派/待办列表中的工单。
3. **分流决策**（二选一或组合，以实现为准）：
   - **分派**：选择目标 **街道**（**龙山街道、雉城街道、画溪街道、太湖街道** 四者**之一**，下拉**单选**；**管理侧分派/转交选项不含「其他」**）。居民上报若选「其他」，分派时仍须映射到四街道之一由管理员人工确认或规则约定）。确认后工单进入该街道待办，并向 **该街道管理员** 发送 **短信提醒** 处置（见 **3.5**）。
   - **自行办理**：由住建局在本平台内完成办理与办结（与乡镇办理步骤一致：办理意见 + 选填图片）。
4. 工单状态随操作流转（见第 6 节「状态」）。

### 4.3 乡镇/街道办理流程

1. 乡镇管理员登录管理端。
2. 在待办列表中查看 **已分派至本乡镇** 的工单。
3. **办理**：必填 **办理意见**；选填 **是否上传图片**（支持 **多张**）。
4. **办理结束**：提交后工单标记为 **已办结**（或等效终态）。

## 5. 功能需求详述

### 5.1 用户端（手机 · 扫码入口）

#### 5.1.1 入口

- 提供 **二维码**，扫码后打开「问题反馈」页面（URL 或小程序路径由技术方案确定）。

#### 5.1.2 问题反馈表单

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 问题描述 | 是 | 文本，支持合理字数上限（建议 500～2000 字可配置） |
| 所在街道 | 是 | **下拉选择**，选项为：**龙山街道**、**雉城街道**、**画溪街道**、**太湖街道**、**其他**；选择 **其他** 时 **无需** 补充填写具体位置 |
| 小区名称 | 是 | 文本 |
| 姓名 | 否 | 文本 |
| 电话 | 否 | 手机号格式校验（若填写） |
| 图片 | 否 | 支持 **多张** 上传；展示与交互采用 **移动端常见样式**（如方形缩略图 +「添加」按钮）；格式与大小限制由技术方案规定 |

#### 5.1.3 提交与反馈

- **提交**：校验必填项通过后提交；失败时提示原因。
- **提交成功**：明确成功提示（无需强制登录）。

---

### 5.2 住建局管理员 · 分流与自办

- 登录管理端（手机或电脑）。
- **待分派/待办列表**：展示未办结且需住建局处理的工单（含待分派、自办中等，列表筛选以实现为准）。
- **分派**：选择 **四个街道之一**（龙山街道、雉城街道、画溪街道、太湖街道，全称），确认后工单归属该街道管理员待办。
- **自行办理**：进入办理界面，填写 **办理意见**（必填）；选填 **图片**（多张）；提交后 **办理结束**。
- **历史记录**：可查阅已分派、已自办办结的工单（字段与导出需求以二期为准）。

---

### 5.3 乡镇/街道管理员 · 办理与办结

- 登录管理端。
- **待办列表**：仅展示 **已分派至本乡镇** 且未办结的工单。
- **办理页**：
  - 展示居民填报信息（问题描述、街道、小区、选填联系方式与图片等）。
  - **办理意见**：必填。
  - **图片**：选填，支持 **多张**（与住建局自办一致）。
- **办理结束**：提交后工单 **办结**，不可再编辑（若需撤回流程，后续迭代）。

---

### 5.4 管理员手机端（与电脑端功能对齐）

- **登录**：5 个账号之一，按角色进入对应菜单与数据范围。
- **功能**：与下述「电脑端」在 **业务操作** 上保持一致——分流、分派、自行办理、乡镇办理、办结、基础列表与详情查看。
- **差异**：布局与交互适配手机屏幕；**数据统计**可在手机端做 **简化展示或仅入口跳转至电脑端**，以实现为准（建议在 PRD 评审时二选一定案）。

---

### 5.5 管理员电脑端（业务 + 数据统计）

#### 5.5.1 业务功能

与 **管理员手机端** 相同：**登录、分流、分派、办理、办结、列表与详情**。

#### 5.5.2 数据统计（电脑端增强）

在电脑端提供 **数据统计/ dashboard**，建议包含以下 **指标与维度**（口径需开发时与业务共同锁定）：

**（1）概览**

| 指标 | 说明 |
|------|------|
| 问题反馈总量 | 平台上报工单累计数（可按时间范围筛选） |
| 已办数量 | 状态为已办结的工单数 |
| 待办数量 | 未办结工单数（可细分：待分派、处理中等，以实现为准） |

**（2）按乡镇/街道分布**

- 各乡镇（及必要时住建局自办）的工单数量分布；建议支持图表（柱状/饼图）与表格。

**（3）按办理时长统计**

从 **进入可办理状态** 至 **办结** 的耗时区间分布（或列表占比），建议区间：

| 区间 | 说明 |
|------|------|
| 24 小时内 | 办结耗时 ≤ 24 小时 |
| 3 天内 | 办结耗时 ≤ 72 小时（或与 24 小时互斥分段，需约定） |
| 7 天内 | 办结耗时 ≤ 168 小时 |
| 超 7 天 | 办结耗时 > 7 天 |

> **注意**：「24 小时 / 3 天 / 7 天」是 **互斥分段** 还是 **累计漏斗**，需在评审时确定一种统计口径，避免重复计数。

**（4）筛选条件（建议）**

- 时间范围：提交时间、办结时间。
- 街道（含龙山/雉城/画溪/太湖/**其他**）。
- 承办主体：乡镇 / 住建局自办。

**（5）导出（可选，二期）**

- Excel 导出：与当前筛选条件一致的数据表。

---

### 5.6 账号管理（管理员电脑端）

- **入口**：电脑端侧栏或主导航中的 **「账号管理」**。
- **列表信息**（建议）：登录账号、姓名、手机号、角色、管辖街道（仅乡镇类角色）、状态、操作（编辑、重置密码等）。
- **新增/编辑**：
  - **角色**：至少区分 **系统管理员** 与 **乡镇/街道管理员**（命名与权限边界以实现为准；系统管理员可维护账号）。
  - **乡镇/街道管理员**：须选择 **一条管辖街道**，单选，选项为 **龙山 / 雉城 / 画溪 / 太湖** 四街道，**不含「其他」**。
  - **登录密码**：新建时必填；编辑时若留空表示不修改；须满足 **5.7.2** 强度要求。
- **重置密码**：触发后向账号 **绑定手机号** 发送短信（临时密码或带令牌的链式重设，实现定稿）。

### 5.7 安全与认证

#### 5.7.1 登录

- 管理端统一使用 **账号 + 密码**；**手机端与电脑端** 同一套认证体系（若单点登录/会话策略不同，在实现阶段说明）。

#### 5.7.2 密码强度

- 用户**新设**或**修改**后的登录密码须同时满足：长度 **不少于 8 位**；且须 **同时包含** 以下四类字符：**大写英文字母、小写英文字母、数字、特殊符号**。

#### 5.7.3 忘记密码

- 提供 **「忘记密码」** 能力：向账号 **已绑定手机** 发送 **短信验证码**，用户输入 **正确验证码** 后，**重新设置** 符合 **5.7.2** 的新密码。
- 验证码有效期、同手机号发送 **频率限制**、每日上限等，与短信平台及等保要求一致，在 **安全设计** 中明确。

#### 5.7.4 输错限制（防暴力破解）

- 在 **短信验证码校验** 与 **密码校验**（含 **登录**、**忘记密码/重设密码** 流程中涉及密码的环节）中：同一账号、同一手机号或同一终端维度（**具体键值由安全评审定**）在短周期内**连续输错超过 3 次**的，**须等待 5 分钟** 后方可再次尝试；锁定时长与计数重置规则在实现中与安全策略一致。

## 6. 工单节点（建议）

详情页时间轴及业务侧展示，建议统一为以下 **四级节点**（顺序固定）：

| 节点 | 说明 |
|------|------|
| **用户提交** | 居民已完成问题反馈表单提交 |
| **待分派** | 住建局尚未完成分派（指派至四街道之一，或进入自行办理前） |
| **处理中** | 责任主体正在办理（街道处置或住建局自办等） |
| **办结** | 办理结束并结案 |

实现层可映射为内部状态码；管理端列表筛选（如待分派 / 待办 / 已办）与上述节点的组合关系以实现为准。

## 7. 非功能需求（摘要）

| 类别 | 要求 |
|------|------|
| 安全 | 管理端强制登录；**密码按强度策略校验与存储加密**；**支持短信验证码的忘记密码/重设密码**；**分派/验证类操作防暴力（如连续错 3 次锁 5 分钟，细则见 5.7.4）**；HTTPS；接口鉴权 |
| 可用性 | 用户端弱网可重试提交；图片上传失败有明确提示 |
| 兼容性 | 用户端与管理手机端适配主流移动浏览器；管理电脑端适配主流桌面浏览器 |
| 性能 | 列表分页；统计查询响应时间目标在实现阶段约定 |
| 合规 | 个人信息（姓名、电话）按最小必要原则存储与展示；隐私政策入口视上线要求补充 |

## 11. 附录：交互原型工作流（工程维护 · 非需求条款）

以下约定用于在本仓库内维护 **交互原型** 与 **静态发布**，**不构成对产品功能或范围的合同式需求**；产品需求以前文 §1～§10 为准。

### 11.1 左侧原型列表（站点地图）

- **位置**：`index.html` 左侧 `.sitemap`，按「用户端 → 管理端手机 → 管理端电脑 → 宣传物料」等分区，每一项对应一个可跳转的 **屏幕块**（`.screen`）。
- **跳转**：链接使用 `href="#屏幕id"`，由 `prototype.js` 中 `navigateTo` / `showScreen` 切换 **仅激活对应 `#id` 的 `.screen`**，并更新地址栏 hash。
- **约定**：每屏一个稳定 **`id`**（如 `m-list-zjj`、`pc-list`）；需「带状态进入」的详情用 **独立函数**（如 `goPcDetail('done')`）在进入前设好状态再 `showScreen`。

### 11.2 多设备预览（手机 / 电脑 / 广告牌）

- **容器**：`#preview-phone-wrap`（手机框）、`#preview-pc-wrap`（宽屏 chrome）、`#preview-billboard-wrap`（5:10 竖版宣传海报，深蓝科技风，素材见 `assets/billboard-bg.png`）。
- **切换**：侧栏链接 **`data-device="phone" | "pc" | "billboard"`**，在 `prototype.js` 的 `setDevice` 中控制显示哪一个预览区。
- **样式**：集中在 **`prototype.css`**；手机与 PC 差异优先用 **class** 或 **媒体查询**，少复制整段 HTML。

### 11.3 行为脚本（`prototype.js`）

- 导航统一走 `navigateTo`；特殊屏在 `navigateTo` 内 **分支初始化**，保证从侧栏进入状态正确。
- 表单校验保持可读函数，避免与导航逻辑混写。

### 11.4 同步到静态站点（博客）

- 将本 **`FeedbackPlatform`** 目录全量覆盖到博客仓库 **`PR/FeedbackPlatform/`**（与其它项目一致置于 `PR/` 下）。
- 在博客仓库根目录：`git add` → `commit` → `push origin main`。
